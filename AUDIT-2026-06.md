# Complete System Audit — GenWatch + ATS-Pi Companion

**Date:** 2026-06-28
**Scope:** Full audit of both projects and their interoperability, for deployment on a
real Generac H-100 generator + ASCO Series 300 transfer switch.
**Hardware on hand:** 2× Raspberry Pi, ADAM-6060 relay module, Waveshare USB-to-RS485 adapter, DB9 adapters.
**Method:** Six parallel deep-audit passes (backend, Modbus/frontend, ATS-Pi firmware,
ICD interoperability, deployment/hardware, and a test-suite run + coverage pass).

## 1. Executive verdict

This is a **well-engineered, defensively-coded foundation** — substantially stronger than
typical for this class of system. The hard problems are mostly already solved:

- **Interoperability is solid.** The ATS-Pi ↔ GenWatch Modbus wire contract (ICD v1.0) was
  cross-checked register-by-register on both sides: **every** read register (0x0000–0x0043),
  write register (0x0100–0x0103), enum, bitfield mask, and u32 word-ordering matches the ICD
  *and* matches between the two implementations. Transport, framer (MBAP/socket — correctly
  *not* RTU-over-TCP), port (5020), unit ID (1), and function codes all conform. **No
  CRITICAL or HIGH interop discrepancies.**
- **Tests are green and honest.** 511 tests pass (245 GenWatch + 266 ATS-Pi), 0 fail / 0 error
  / 0 skip; both projects lint clean (ruff). The ATS-Pi contract test drives a *real* pymodbus
  wire, the safety-watchdog tests fail *real* writes rather than asserting mocks, and known
  spec deviations are tested honestly rather than papered over.
- **Safety architecture is real:** comms-loss auto-release, panel-AUTO command gates, two-step
  confirm tokens, last-good register preservation (no sentinel-zero injection), production
  secret refuse-to-boot, atomic register-block swaps, persistence via temp+rename+fsync, and a
  layered relay fail-safe (software → ADAM FSV → cable-pull).

**However — before trusting this on a live 600 A switch, there is one CRITICAL safety gap and a
cluster of HIGH items that must be addressed.** They are listed below in priority order. None
require a redesign; all are bounded fixes.

## 2. Findings by priority

Severity key: **CRITICAL** = can mis-operate hardware or strand it in an unsafe state /
data loss / security breach · **HIGH** = wrong operator-facing data or a defeated safety
layer under plausible conditions · **MEDIUM** = degraded reliability/serviceability ·
**LOW/INFO** = hardening.

### CRITICAL

**C-1 · ATS-Pi comms-loss auto-release can be defeated by *any* Modbus reader.**
`ats-pi-companion-main/src/atspi/server.py:151-153`, `safety.py:41-49`
The §8.3 safety contract ("if no successful read *from GenWatch* for 30 s, auto-release
inhibit + force-transfer") is implemented as "any successful read from *any* client refreshes
the timer." A diagnostic `modpoll` loop, a Prometheus modbus exporter, a network scanner, or a
**stale/second GenWatch instance** polling the ATS-Pi will indefinitely suppress the
auto-release. If GenWatch proper dies after asserting `force_transfer=1`, the 600 A switch can
stay forced to emergency with no automatic release. This defeats the single most important
safety guarantee in the ICD.
*Fix:* Key the watchdog on the authoritative GenWatch connection only — either count only reads
of the core-state block from the identified peer, or require GenWatch to write a heartbeat
register and arm the timer off that.

### HIGH

**H-1 · ATS-Pi crash-time backstop is weak and the waiver is too easy.**
`ats-pi-companion-main/src/atspi/config.py:43-49`, `io_adam.py:506-577`; deployment:
no `RuntimeWatchdogSec` drop-in, `systemd/atspi.service`.
The software watchdog shares the process's fate, the hardware ADAM host-watchdog registers are
unconfigured by default, and `require_hw_watchdog: false` is a one-line waiver. With the waiver
set, a kernel panic / brownout / OOM-kill while force-transfer is latched has **no** automatic
release. The ATS-Pi — the more safety-critical of the two Pis — also ships *without* the
Pi-level hardware watchdog that GenWatch has.
*Fix:* Ship the `10-*-hwwatchdog.conf` drop-in for the ATS-Pi too; make the `false` waiver
require a second explicit acknowledgement key and refuse to start otherwise; verify the ADAM FSV
cable-pull test as mandatory acceptance.

**H-2 · ATS-Pi serial-sense path can serve a frozen-but-valid position as "fresh & authoritative."**
`ats-pi-companion-main/src/atspi/io_asco_serial.py:227-266`
INPUT_FAULT is raised only when the serial read *throws*. If the Group-5 controller hangs but
the UART still answers (or a register latches), stale position bits are published as fresh,
fault-free, and authoritative — exactly the "reachable but blind" case §10 says must be flagged.
GenWatch would then trust a frozen load source and even permit command asserts it should refuse.
*Fix:* Add a freshness/heartbeat check on the Group-5 data (a counter/status register that must
change, or a cross-field plausibility check) and raise INPUT_FAULT when it stalls.
*(This is also the system's top **test gap** — see §4 — the producer side of "INPUT_FAULT +
serve last-good position" is not asserted end-to-end.)*

**H-3 · GenWatch freshness gate only blocks `comms == "lost"`, not stale "degraded" data.**
`GenWatch-main/backend/genwatch/services/control.py:231-271`, `api/events.py:122-131`
A remote start/stop is gated on `panel_mode == auto` and `engine_state ∈ allowed`, but those can
be served from cached/last-known registers while the link is "degraded" (intermittent prime
polls) rather than fully "lost." The gate the code comment claims defends against "minutes-old
data" only fires on total loss. A stop issued against a stale `running`/`alarm` state, or a start
authorized while the panel was locally switched out of AUTO, is the failure mode these gates
exist to prevent.
*Fix:* Gate on the *age* of the specific key-switch/engine-state registers (the poller already
tracks `value_ages`), or require `comms.state == "healthy"` on the control path.

**H-4 · `h100.yaml` register scale errors produce 10×-wrong operator readings.**
`GenWatch-main/backend/genwatch/registers/h100.yaml:249, 226`
Cross-checked against the genmon `HPanelReg` reference this map cites:
- `run_hours` (0x0130) is missing the `/10` divider → **engine hours read 10× too high**
  (breaks oil-change/warranty hour tracking).
- `coolant_level` (0x0090) has a spurious `scale: 0.1` → **reads 10× too low** → either a
  permanent false "low coolant" nuisance or a real low-coolant state scaled into ambiguity.
*Fix:* `run_hours` → `scale: 0.1`; remove `scale: 0.1` from `coolant_level` (confirm raw
engineering unit on the panel). Update the mock to feed raw ×10 values so the decode path is
actually exercised.

**H-5 · `alarm_range` / `warn_range` are dead config — no numeric alarm comparator exists.**
`GenWatch-main/backend/genwatch/registers/h100.yaml:224-245`, `modbus/registers.py:399-400`,
absent from `services/state.py`
The YAML lets an operator set `alarm_range: [120, 235]` on coolant temp (etc.), and the fields
are parsed — but **nothing compares a decoded value against them.** All alarms derive solely
from the controller's status bitfields. An operator reasonably believes they've armed a software
over-temp/over-pressure backstop; they have not.
*Fix:* Either implement a debounced numeric range comparator in `state.py` (skipping `None`/
stale values, gated on engine-running), or remove the ranges and rename to a clearly
display-only key. Do not ship safety-suggestive config that does nothing.

**H-6 · ATS-Pi RS-485 adapter uses raw `/dev/ttyUSB0` with no stable device path.**
`ats-pi-companion-main/config.example.yaml:138`, `config.py:92`; no udev rule in the repo.
`/dev/ttyUSBn` is assigned in enumeration order — after a reboot or re-plug the Waveshare
adapter can become `ttyUSB1`, the hybrid driver loses ASCO position sensing, raises INPUT_FAULT,
and GenWatch silently falls back to the less-accurate H-100-derived load source. GenWatch solved
this exact problem with `99-genwatch-modbus.rules`; the ATS-Pi shipped nothing equivalent.
**Given you have the Waveshare adapter in hand, this is the single most likely real-world
failure.**
*Fix:* Ship a udev rule creating `/dev/atspi-asco` keyed on the adapter's VID:PID
(CH340 `1a86:7523`) or `ID_SERIAL_SHORT`, install it from `install.sh`, and default the config
to the symlink.

**H-7 · ATS-Pi dependencies are unpinned; no lockfile; live-network install.**
`ats-pi-companion-main/pyproject.toml:26-37`, `install.sh:75-80`
`install.sh` does `pip install "$REPO_DIR"` against live PyPI with `>=` ranges and no hashes.
The two Pis can diverge, a re-image months later pulls different wheels, and the install assumes
internet at a (often air-gapped) generator site. GenWatch, by contrast, ships a sha256-pinned
`requirements.lock`. (Verified live: the ATS-Pi dev tooling resolved to pytest 9 / ruff 0.15,
newer than its documented baseline — still green, but a latent reproducibility risk.)
*Fix:* Generate a hash-pinned lockfile (`pip-compile --generate-hashes`), install with
`--require-hashes`, document an offline wheelhouse path.

**H-8 · GenWatch never forwards ATS-side alarms to Slack.**
`GenWatch-main/backend/genwatch/services/ats.py` (`_forward_to_slack` has no `alarm` branch),
`main.py:377-382`
ATS faults (`ATS_PI_INPUT_FAULT`, `CALIBRATION`, `TIME_SKEW`) raise DB alarms and publish to the
bus, but only `ats-position`/`ats-comms` are forwarded to Slack. A stuck ATS aux contact or
impossible-position fault — precisely what makes the displayed load source wrong — generates
**no push notification**, even with alerting enabled.
*Fix:* Add an `alarm`/`alarm-cleared` branch to the ATS Slack forwarder, mirroring the H-100 path.

### MEDIUM (summary — details in the per-area sections below)

- **M-1** ATS-Pi `systemd/atspi.service`: `MemoryDenyWriteExecute=true` can SIGSYS-kill the
  switch-command service on some CPython builds; GenWatch deliberately omits it. Validate on the
  target image or drop it.
- **M-2** ATS-Pi unit lacks restart-loop protection (`StartLimitIntervalSec`), OOM bias, and
  resource caps that GenWatch has → can latch in `failed` requiring on-site `reset-failed`.
- **M-3** Neither installer enforces/verifies NTP, despite the hard ICD <5 s skew requirement and
  the Pi's lack of an RTC → bogus timestamps + `TIME_SKEW` after a site power cut.
- **M-4** ATS-Pi lifetime transfer count is bumped in memory before the async fsync → a power cut
  in that window makes the count go *backwards* on reboot (violates §8.2 monotonicity).
- **M-5** On the serial path, transfer counting depends on an optional `transferring_bit`; if
  unset, `utility → unknown → generator` never increments the counter or stamps timestamps.
- **M-6** GenWatch DB: event-loop-thread audit/alarm writes share one connection with worker-thread
  telemetry writes under an RLock; a `synchronous=FULL` fsync can stall safety-relevant audit
  writes. Consider routing them through a thread or bounding lock-hold time.
- **M-7** GenWatch login rate-limit + audit IP key on `request.client.host`, which collapses to
  `127.0.0.1` behind the loopback proxy → one attacker can lock out all operators and forensic
  attribution is lost.
- **M-8** GenWatch CSRF middleware + WS origin check fail open when both Origin and Referer are
  absent; non-control config/slack/reload endpoints rely on it.
- **M-9** Hourly rollup computes kWh as `SUM(kw_avg)/60` and averages-of-averages → energy/history
  under-reports with any data gaps. Weight by `samples`.
- **M-10** Malformed-value command writes (e.g. `cmd_inhibit=5`) are ACKed-then-silently-dropped by
  the ATS-Pi rather than rejected (pymodbus `validate()` can't see the value). Unreachable from
  the real GenWatch consumer (which only sends 0x0000/0x0001) but worth tightening.
- **M-11** `battery_volts` warn band in the YAML is 12 V; the 12 V/24 V auto-band logic lives only
  in the frontend, so Settings/any-future-comparator see the wrong band on a 24 V bank.
- **M-12** ADAM `di_read`/`do` function-code mapping is a documented guess (FC01 vs DI); a wrong
  guess yields permanent `position=unknown` (safe, but degraded). Commissioning must verify.

### LOW / INFO

Notable positives and minor items (full list in the appendix per area): correct
endianness/framer selection on both sides; per-register TTL eviction with no phantom-fresh data;
systemd watchdog withholds `WATCHDOG=1` on a wedged poller using monotonic time; pulse-release
retries until it lands and records commanded-OFF *before* the write so a stranded relay still
raises OUTPUT_FAULT; persistence double-fsync + temp+rename; secrets refuse-to-boot on both
sides; ATS-Pi config rejects unknown keys (GenWatch silently ignores them — consider
`extra="forbid"`); the two test-doubles (GenWatch's `mock_ats_pi.py`) are *more permissive* than
the real ATS-Pi server (no write-mode gating, mirrors written value instead of driven state) —
add a guarded context or a GenWatch-side address-conformance test so a `ats_pi.yaml` typo can't
slip through.

## 3. Interoperability conformance — PASS

Register-by-register cross-check of ICD ⇄ GenWatch consumer ⇄ ATS-Pi producer:

| Area | Result |
|---|---|
| Transport (Modbus TCP / MBAP socket framer, port 5020, unit 1, FC03/FC06/FC16) | ✅ Match on both sides |
| Core state 0x0000–0x0005 (position/avail/start/mode/fault enums + masks) | ✅ Identical both sides |
| Timestamps/counters 0x0010–0x0023 (u32, high-word-at-low-addr) | ✅ Word order agrees |
| Identification/version 0x0030–0x0035 | ✅ ICD v1.0, unit_id 23 wired |
| Command read-back 0x0040–0x0043 (driven state) | ✅ Match |
| Command writes 0x0100–0x0103 + read-back mapping (0x0100+N → 0x0040+N) | ✅ Match |
| Semantic contracts: 30 s auto-release, no-silent-reassert, INPUT_FAULT/CALIBRATION drop-authority, version mismatch, TIME_SKEW, reboot detect, §10 precedence | ✅ Implemented on the correct side and agree |

The only interop-adjacent caveats are **C-1** (the auto-release *trigger* is too broad — a
semantic gap, not a wire mismatch) and the advisory pymodbus exception-code deviation (0x02 vs
the ICD's preferred 0x03/0x04), which the ICD explicitly tolerates because GenWatch ignores the
code.

## 5. Recommended remediation order

1. **C-1** — scope the ATS-Pi auto-release watchdog to GenWatch's connection. *(safety, blocking)*
2. **H-6** — udev symlink for the Waveshare RS-485 adapter. *(you have this hardware; will bite first)*
3. **H-1** — ATS-Pi hardware watchdog + harden the `require_hw_watchdog` waiver.
4. **H-3** — tighten GenWatch's control freshness gate to register age, not just `comms==lost`.
5. **H-4 / H-5** — fix the two `h100.yaml` scale errors; resolve the dead range config.
6. **H-2 / test gap** — serial-sense staleness detection + the end-to-end INPUT_FAULT test.
7. **H-7 / H-8** — pin ATS-Pi deps; forward ATS alarms to Slack.
8. MEDIUM items (NTP enforcement, restart-loop protection, MDWE validation, counter durability,
   rollup math, rate-limit IP) — before unattended field service.

The foundation is sound. Closing C-1 and the HIGH cluster makes it genuinely field-ready.
